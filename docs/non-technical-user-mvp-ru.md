# Спецификация: Mnemos для не-IT пользователя

Этот документ описывает, какого функционала сейчас не хватает, чтобы
Mnemos можно было давать обычному пользователю без API, CLI и ручной
работы в Terminal.

Сейчас ядро системы уже есть:

- хранение memory items
- semantic search
- ingestion
- fact extraction
- reflection generation
- candidate governance

Но отсутствует продуктовый слой, который делает всё это понятным и
удобным для не-технического человека.

---

## Цель

Сделать так, чтобы человек с MacBook и Docker Desktop мог:

1. установить Mnemos по пошаговой инструкции
1. открыть понятный веб-интерфейс
1. загрузить свои данные без работы с JSON, YAML и `curl`
1. искать по памяти обычным текстом
1. видеть разницу между заметками, фактами и reflections
1. принимать или отклонять новые кандидаты кнопками

---

## Ключевая проблема текущей версии

Сейчас Mnemos можно запустить локально, но им нельзя нормально
пользоваться без технического посредника.

Почему:

- нет стартовой веб-страницы
- нет onboarding wizard
- нет экрана импорта данных
- нет формы ручного добавления записей
- нет поискового экрана
- нет экрана review кандидатов
- нет пользовательских сообщений об ошибках
- нет встроенной справки

---

## Что нужно реализовать

## 1. Веб-приложение поверх существующего API

Нужен отдельный user-facing web app, который работает поверх уже
существующих endpoints.

### Минимальные разделы

- Главная
- Поиск
- Память
- Импорт
- Кандидаты на проверку
- Настройки
- Справка

### Критерии готовности

- пользователь может открыть интерфейс в браузере
- основные действия доступны без Terminal
- во всех основных сценариях есть кнопки и формы вместо `curl`

---

## 2. Первый запуск через setup wizard

После первого открытия пользователь должен видеть не пустой API-экран,
а понятный мастер запуска.

### Wizard должен уметь

- проверить, что API отвечает
- проверить готовность PostgreSQL и Qdrant
- объяснить, что уже готово, а что нет
- предложить "Начать с демо-данных" или "Загрузить свои данные"
- показать понятный статус: "система готова", "нужна настройка",
  "ошибка запуска"

### Экран должен отвечать на вопросы

- что такое Mnemos
- что можно сделать прямо сейчас
- сколько шагов осталось до первого результата

---

## 3. Импорт без файлового формата в голове

Сейчас ingestion требует подготовить Markdown, YAML или JSONL. Для
обычного пользователя это слишком высокий порог.

### Функции импорта

- форму "Вставить текст"
- загрузку файла через drag and drop
- визуальный выбор типа данных
- заметки
- ответы опросника
- список наблюдений
- предварительный просмотр того, что будет импортировано
- понятный отчёт об импорте
- сколько записей добавлено
- сколько пропущено как дубликаты
- что не удалось разобрать

### Критерии готовности импорта

- пользователь может загрузить данные, не зная JSONL и YAML
- перед импортом видно, что именно попадёт в систему
- ошибки описаны простым языком

---

## 4. Экран поиска и просмотра памяти

Пользователь должен иметь возможность искать память как в обычном
приложении.

### На экране поиска должны быть

- строка поиска
- фильтр по domain
- фильтр по типу записи
- список результатов
- карточка результата с понятными полями

### В карточке записи должны быть

- основной текст записи
- тип записи
- область памяти
- дата создания
- confidence, если он есть
- связи с evidence, если они есть

### Критерии готовности поиска

- пользователь может открыть поиск и сразу начать вводить вопрос
- результаты можно читать без знания внутренних терминов
- типы записей объяснены прямо в интерфейсе

---

## 5. Экран review кандидатов

Сейчас review выполняется через API и CLI. Для обычного пользователя
нужен отдельный экран модерации.

### Экран должен показывать

- список pending candidates
- текст кандидата
- тип кандидата
- confidence
- evidence
- возможные validation errors

### Действия

- принять
- отклонить
- отклонить с причиной
- отложить на потом

### Критерии готовности review

- review можно выполнить без UUID и terminal-команд
- пользователь понимает, что именно он принимает
- видно, на каких evidence опирается запись

---

## 6. Ручное добавление записи через форму

Нужна простая форма "Добавить запись".

### Поля формы

- область памяти
- тип записи
- текст записи
- уверенность, если нужно
- источник или комментарий

### Дополнительные элементы формы

- быстрые шаблоны
- заметка о себе
- наблюдение о проекте
- решение
- задача

### Критерии готовности формы

- новую запись можно добавить за 30-60 секунд
- пользователь не обязан знать структуру API payload

---

## 7. Человеческие статусы и сообщения

Сейчас сообщения системы ориентированы на технический слой. Нужны
нормальные пользовательские статусы.

### Нужные сообщения и состояния

- пустые состояния
- подтверждения успешных действий
- дружелюбные ошибки
- подсказки "что делать дальше"

### Примеры

Хорошо:

```text
Система готова к работе

Теперь вы можете добавить первую заметку или загрузить файл с ответами.
```

Плохо:

```text
HTTP 500
```

---

## 8. Встроенная помощь

Внутри будущего интерфейса нужна встроенная справка.

### Разделы встроенной помощи

- экран "Что такое Mnemos"
- экран "Как добавить первую память"
- экран "Как устроены facts и reflections"
- экран "Что такое кандидат и зачем нужен review"

### Критерии готовности помощи

- пользователь может разобраться без чтения исходников репозитория
- терминология объяснена прямо в продукте

---

## 9. Резервное копирование и экспорт

Для обычного пользователя важно понимать, что его данные можно вынести
наружу и сохранить.

### Возможности экспорта

- экспорт памяти в читаемый формат
- резервное копирование настроек
- кнопка "Скачать мои данные"

### Почему это важно

Без экспорта обычный пользователь не доверяет системе как личной памяти.

---

## 10. Ясная модель "что это за запись"

В интерфейсе нужно убрать внутреннюю сложность терминов.

### Пользовательская подача

- `raw` = исходная запись
- `fact` = проверенный факт
- `reflection` = общий вывод
- `candidate` = новое предложение на проверку

### Требование

Во всех местах интерфейса должен быть либо перевод этих терминов в
понятный язык, либо встроенная подсказка.

---

## Приоритет реализации

### Этап 1. Без этого не будет non-technical UX

- веб-приложение
- setup wizard
- экран поиска
- форма добавления записи
- экран review кандидатов

### Этап 2. Делает продукт реально удобным

- импорт через UI
- человеко-понятные статусы и ошибки
- встроенная помощь

### Этап 3. Делает продукт зрелым

- экспорт и резервные копии
- onboarding с демо-режимом
- пользовательские шаблоны и guided flows

---

## Acceptance Criteria для MVP

Можно считать, что non-technical MVP готов, если пользователь без
опыта работы с API способен:

1. открыть приложение в браузере
1. понять за 2-3 минуты, что делает Mnemos
1. добавить первую запись без `curl`
1. найти её через поиск
1. загрузить файл без подготовки YAML или JSONL
1. принять или отклонить candidate кнопкой

---

## Что уже можно переиспользовать из текущего ядра

Будущий продуктовый слой не нужно писать с нуля по данным и логике.
Можно опираться на уже существующее ядро:

- `POST /memory/items`
- `GET /memory/item/{id}`
- `POST /memory/query`
- `POST /memory/candidate`
- `GET /memory/candidates`
- `POST /memory/candidate/{id}/accept`
- `POST /memory/candidate/{id}/reject`
- ingestion, extraction и reflection pipelines

То есть сейчас недостаёт не memory engine, а пользовательского продукта
поверх него.
